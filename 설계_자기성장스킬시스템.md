# 설계 — 자기성장 스킬/지식 시스템 (v0.6)

> 대화·피드백을 **에이전트의 판단**으로 스킬/지식에 누적해 회귀 없이 성장시키는 루프.
> v0.5 → v0.6 (3차 크로스체크 반영): 기본 등급 PII 배포누출 차단(사이드카+source_quote 정제+배포 사전스캔), update_count 폐기→파생 성숙도, 지식 탐지레이어(claim 구조+반박 깃발+conflict 게이트+must 금지), P0 선결(카운터 frontmatter 분리·skill_id 레지스트리).
> 변경/신규는 `[v4]`/`[v5]`/`[v6]` 표시. **정합성 판정: 조건부 GO — 위 P0 선결 반영 완료, P0 잠금 가능.**

## 배포 경계 (사실, `.gitignore` 기준) `[v5]`
- `도구/스킬/자산/지식`은 **`기본`만 GitHub에 올라간다.** `추가·고급·대외비`는 **로컬 전용(배포 제외).**
- **모든 등급은 내부에서 검색·사용된다(대외비 포함).** 등급은 *접근 권한*이 아니라 *배포 여부*를 가른다.
- 대외비 = "개인/회사 식별정보가 들어가서 배포할 때 빠져야 하는 것." 발견 필터 같은 건 없다.

## 0. 불변 원칙

- **P0 (신규, 최상위). 발의(열기) ≠ 커밋(게이트).** `[v4]`
  - **누구나** 스킬/지식 변경을 *발의*(candidate 생성)할 수 있다(진입장벽 폐지 = D1).
  - 그러나 전역 자원으로의 *커밋*은 호출자와 무관하게 **콘텐츠 게이트 5종**(P1 판단계약·민감도·자원락·버전·non_overridable)을 항상 통과한다.
  - **identity는 "쓰기 자격"이 아니라 "자동승인 여부"만 결정한다.** 대표 발의 = 자동승인(즉시 적용). 그 외 = candidate로 수렴 절차.
- **P1. 의미 변경은 에이전트, 가드레일은 기계.** 케이스/스킬의 모든 의미 변경(등록·주입선별·강등·supersede·통합·폐기·신규생성·등급/민감도/라우팅 분류)은 에이전트가 읽고 판단한 산출물이 있을 때만. 키워드·정규식·카운터는 어느 단계서도 의미변경 트리거 불가 — *신호*(정렬·깃발·검토큐·격리)로만.
- **P2. 판단 주체 분업.** `[v4]` doer(작업) ≠ 공간관리(평가·후보 발의) ≠ 관리자(전역 commit 승인). **별도 "스킬 steward" 에이전트는 신설하지 않는다**(자기정당화 편향은 이 분업으로 이미 차단; 신설은 wake 제약·락 경합만 키움).
- **P3 (재정의). 본문 = 조건무관 범용 절차 / 케이스 = 조건부 델타.** `[v4]`
  - 본문은 **덮어쓰기 가능(D2)**, 단 *버전드 개정*(이전 본문 스냅샷 보존 → 롤백 가능)으로만. 화면은 덮어쓰기, 저장은 새 버전.
  - 본문엔 특정 인물·숫자·1회성을 박지 않는다(그건 케이스로). 본문 개정은 **여러 케이스가 보편패턴으로 수렴했을 때만**(단일 케이스발 본문개정 금지).
  - `non_overridable`(케이스·개정으로도 못 지우는 안전지침) 섹션 유지. 본문 개정 diff가 이 섹션을 약화하면 거부.
- **P4. 범용 스킬 + 케이스.** 신규 남발 금지(§4 비차단 점검).
- **P5 (재정의). 등급 = 배포 여부 / 민감도 = 보수적 기본값.** `[v5]`
  - 등급은 *배포 경계*를 가른다(기본=배포, 추가·고급·대외비=로컬). **접근 차단 아님 — 모든 등급 내부 검색·사용.**
  - 개인/회사 식별정보 *가능성* → 일단 **대외비**(=배포에서 빠짐). 내부 사용엔 제약 없음.
  - **대외비→기본 승격(declassify=배포대상화)은 대표 확인 필수**(배포 노출이 새로 생기므로). 그 반대(기본→대외비, 더 안전)는 누구나 발의 가능.

## 1. 기존 LessonLedger와의 관계 — 레슨은 "임시 분류 버퍼"로 축소 `[v4]`

채택: **역할 분리 + 레슨 영구저장소 역할 제거.**
- **레슨 = 공간 내부·검증 전·휘발성 교정 신호 버퍼.** 자동 캡처(평가)·growth_gaps(작업큐)·주입은 **유지**(유용한 단기 인프라).
- **영구 학습은 레슨에 쌓지 않고 졸업시킨다:** 절차적("어떻게") → **스킬 케이스**, 사실/맥락/선호 → **지식 본문 개정**. 수렴 도달 시 레슨은 `graduated/superseded`(삭제 아님).
- **스킬/지식 라우팅 판단은 자동화 금지(P1)** — "레슨 전부 자동 지식화"는 law "지식=참고/스킬=사용" 위반. 에이전트가 polarity·근거 읽고 명시 결정.
- 옵션 비교 결론: 완전폐기 ✗(발행 게이트·주입·테스트 251줄 붕괴), 현행유지 ✗(이중 영구저장소·조용히 썩는 must), **버퍼 축소 ✓.**

## 2. 핵심 모델 — 레코드

```
스킬/{등급}/{이름}/
  ├ SKILL.md            ← 범용 본문 + non_overridable + frontmatter(불변/저빈도 필드만)
  ├ cases.jsonl         ← 공개안전 케이스(append-only). 기본 등급이면 이 파일만 배포
  ├ cases.local.jsonl   ← 식별정보 포함 케이스 + 원문 raw audit (항상 gitignore=배포제외) [v6]
  ├ cases.archive.jsonl ← graduated/superseded/expired 컴팩션 (gitignore) [v6]
  └ case_events.jsonl   ← 적용 결과/카운터 이벤트(append-only, gitignore)
```
SKILL.md frontmatter — **불변/저빈도 필드만**(배포 diff·롤백 오염 방지) `[v6]`:
```yaml
skill_id: 안정식별자(경로/등급 무관 불변)   # 모든 참조는 이 id로. 케이스에 경로필드 금지
name / description: 발견용(계속 갱신, D3)
version: 본문 버전(본문 개정 시만 +1, CAS)
last_updated: 마지막 본문개정 시각
```
- **성숙도 카운터는 frontmatter에 두지 않는다(`[v6]`).** `update_count` 같은 단일 합산수치는 폐기 — 나쁜 케이스도 +1되어 품질을 호도한다. 성숙도는 cases.jsonl/case_events에서 **파생 계산**해 발견 시 노출(§5): `worked_ratio`, `harmful>0 경고`(1건도 신호), `stability`(최근 개정 간격). 합산 금지, 분해 노출.
- **skill_id↔현재 경로 매핑은 발견기 인덱스를 단일 진실원으로**(등급 재분류해도 참조 불변). cases.jsonl은 스킬 폴더에 동거하므로 이동 시 함께 따라감.
- **draft 개념 없음.** 신규도 실 스킬. "신규(미검증)" 표시는 `cases==0`에서 **파생**(수동 라벨 아님 → P1 정합).
케이스 레코드(판단 계약 필수):
```jsonc
{
  "case_id":"...", "skill_id":"...",
  "scope":"global|space", "origin_space":"...",
  "condition":"자연어(키워드 게이트 없음)",
  "instruction":"역할/패턴으로 일반화, 구체 PII 금지",
  "polarity":"worked|failed",                  // 필수
  "action":"add_case|supersede|new_skill|none",  // 필수 [v6: draft 용어 제거]
  "routing_kind":"procedural|factual|preference",  // 필수 [v6] 스킬케이스 vs 지식 라우팅 명시
  "judgment_rationale":"...",                  // 필수
  "source_quote":"근거 발화의 PII 정제 요약",   // 필수 [v6] 원문 1:1 아님. raw 원문은 cases.local.jsonl 전용
  "sensitivity":"public|confidential",          // 미판단/불확실 = confidential(보수적 기본값) [v6]
  "evidence_level":"user_directive|verified_result|agent_observation|reviewer_approval",
  "status":"candidate|provisional_must|active|superseded|retired|conflict|graduated|expired",
  "review_due_at":"provisional_must만 의무",
  "supersedes":"이전 case_id|null",
  "proposed_by":"발의자(채팅/작업/공간관리/관리자/대표)",   // [v4] 누구나 발의
  "approved_by":"승인자(대표=자동 / 관리자 / null)"          // [v4] 발의≠승인 분리
}
```
- SKILL.md frontmatter에 **`version`**: 본문 개정은 `expected_version` 낙관적 락(CAS)으로 stale write 거부. `[v4]`

## 3. 판단 루프 (P1)
피드백/결과 발생 시 판단 주체(공간관리/관리자)가 추론으로:
1. **이해·분류** — polarity + source_quote(근거 문장) 명시.
2. **모순 점검(등록 전 강제 read)** — 같은 skill_id + 겹치는 condition active 케이스 주입받아 비교 → 신규/보완/supersede/conflict 택1(미선택 거부).
3. **민감도 자기점검** — 식별정보 있나? → 일반화 + sensitivity. **권한과 독립한 계약 필드(없으면 거부).** `instruction`뿐 아니라 **`source_quote`·`judgment_rationale`도 일반화 대상**(특히 기본 등급 향). confidential이면 cases.local.jsonl로 라우팅. `[v6]`
4. **행동 선택 + 라우팅** — none/add_case/supersede·강등/new_skill. **`routing_kind`(procedural→스킬 케이스 / factual·preference→지식)를 명시**(키워드 아님, 판단). `[v6]`
5. **근거 기록** — judgment_rationale + source_quote(없으면 거부).
→ 등록 함수는 `polarity·action·judgment_rationale·source_quote` 4필드 충족 시만 통과(계약).

## 4. 새 스킬 생성 — 비차단 점검 + 대표 override `[v4]`
자연어 요청(D4)이 오면 매니저가 **LLM 판단으로** "스킬 작업 요청"임을 인식 → request_work로 doer 위임 → doer가 발견기로 **skill-creator를 응용**(본문/신규). **결과물은 `스킬/후보/` 초안**으로만, 전역 배치는 관리자 승인.
- **3중 점검(차단 아님, 자동·제안형):** ①목표를 3표현 재검색 ②반복성 판단 ③생성 후 역방향 중복 플래그. 기존 스킬 있으면 "신규 대신 케이스 추가/본문 보완할까요?" 제안. **대표가 "그래도 새로"면 override**.
- **중복 점검은 어휘 발견기(discovery.py)에 의존 금지(`[v6]`)** — `score()`가 부분문자열 일치라 동의어/패러프레이즈 중복을 못 잡는다. 중복 판정은 **임베딩 유사도 또는 LLM 의미비교**로. (이 한계를 설계에 명시.)
- **신규 스킬은 처음부터 실 스킬**(draft 단계 없음, `[v5]`). 성숙도는 `update_count`/`stats`로 드러냄 — 갓 만든 스킬은 카운터가 낮아 *신선함이 그대로 보임*(차단 아님).
- 기본 등급 = `추가`(=배포는 안 되지만 내부 사용). 식별정보 가능성 → `대외비`. 범용·필수로 검증되면 `기본` 승격(=배포대상화, 대표 확인). **apply 경로에 등급 필수 입력**(현재 `스킬/추가/` 하드코딩 버그 수정).
- 발견 형식 게이트(P1 비위반): 생성 직후 "부를 표현 3개"로 `발견.py` → 자기 스킬 top-3 미진입 시 description 재작성·재검사.

## 5. 사용 시점 주입 (읽기)
- 재사용: build_lesson_pack의 예산/3단/우선순위. **미재사용: `_keyword_matches` 키워드 게이트(P1 위반).** 후보 선별 = scope 매칭 + 임베딩 *정렬 힌트만*, 적용 판단은 에이전트.
- 예산 컷 보호: worked 핵심 케이스 must 보호 + `excluded by budget` 발생 시 대시보드 경고 + Curator 분리 트리거.
- **발견 환류(D3):** 케이스/본문개정으로 트리거 용어가 바뀌면 frontmatter description 갱신(본문 아님, 허용). 본문 버전업 시 §4 발견 형식 게이트 재실행 의무.
- **성숙도 노출(`[v6]`):** 발견·선택 시 **파생 지표**를 보여 신뢰도를 가늠 — `worked_ratio`(방향), `harmful>0`이면 경고(절대값, 1건도 신호), `stability`(최근 개정 간격↓=수렴), `cases==0`이면 "신규". *합산 단일수치(update_count) 쓰지 않음.* 차단 아님, 판단 보조(P1 정합). 모든 등급(대외비 포함) 검색됨.

## 6. 수렴 / 오피드백 안전
- 진입 status는 **출처 종속:** 대표 발=`provisional_must`(즉시 적용·잠정), 그 외=`candidate`. active 확정 = worked N회+부정0 또는 2차 독립판단.
- **미성숙 가드(`[v6]`):** `cases==0`인 신규 스킬은 케이스를 `provisional_must`로 못 올린다. active 확정의 "2차 독립판단" 경로도 **worked≥1 전제.** (성숙도 노출은 *힌트*, 이 게이트가 *안전장치* — 둘을 혼동 금지.)
- **검토큐 폭주 방지(D1 다발 발의 대비):** `[v4]` candidate(비대표)는 review_due 없이 **수렴 실패 시 자동 expire**. `provisional_must`(대표발)만 `review_due_at` 의무 → 기한 도래 시 대시보드 강제 노출. 검토 부담을 대표 지시 건으로 한정.
- harmful = **깃발만**(검토큐 등록, 케이스 불변). 강등/supersede는 에이전트 판단. harmful 신호 2채널(적용 에이전트 자기보고 + 별도 Reflector 사후판단).
- append-only + proposed_by/approved_by + 근거 → 되돌리기. 단 발견돼야 작동하므로 재검토 기한·conflict 노출·budget 경고가 실제 트리거.

## 7. 권한·동시성 — "발의 개방 / 커밋 게이트" `[v4 전면개정]`
- **law_manager의 "수정 권한 오직 관리자"는 *지침 파일·대시보드 코드*로 범위 한정.** 스킬/지식 자원은 아래 신권한모델.
- **발의:** 누구나(채팅/작업/공간관리/관리자/대표) candidate 생성 가능(D1).
- **커밋:**
  - 대표 발 = 자동승인 → 즉시 적용(provisional_must), 단 **직렬화 writer** 통과.
  - 그 외 발 = candidate → 수렴 또는 관리자 승인(apply).
  - doer는 전역 직접 쓰기 금지 → `스킬/후보/` 초안만.
- **락:** `_with_resource_lock(skill_path)` **신설**(현재 공간 단위 락만 존재 — 전역 `스킬/` 무방비). 본문 read-modify-write는 락 + **version CAS**(읽은 version과 쓸 때 일치 확인, 불일치 재판단).
- **관리자 트리거:** growth_gaps의 `resource_change_needed`를 관리자 작업 큐로(상시 상주 아님).

## 8. Curator (P1 정합)
- 결정론 허용 = **완전 동일 케이스 멱등 dedup**까지만. "비슷한" 통합·폐기·강등·스킬분리는 **제안(대시보드)까지만**, commit은 판단/사람확인.
- **본문 개정 = 단일 경로:** `[v4]` 케이스가 보편패턴 수렴 → 본문 개정 후보 깃발 → 개정. (D2의 "범용 종합"과 Curator 경로를 하나로 통합, 이중정책 제거.) 개정 시 **개정 전 케이스 회귀테스트**(기존 active 케이스를 새 본문에 재투영해 conflict 증가 시 차단) + 이전 본문 스냅샷 보존 + diff 노출 + non_overridable 보존검사.

## 9. 확정된 결정 `[v5]`
1. **대외비 발견 필터 = 없음(폐기).** 모든 등급 내부 검색·사용. 대외비는 *배포 제외*일 뿐. 다만 **배포되는 `기본` 자원에 식별정보가 들어가면 안 됨** → 민감도 자기점검(§3.3, 기본 커밋 시 엄격) + cases.local.jsonl 사이드카 격리 + 배포 전 사전스캔(§10-7,8) 3중 방어.
   - **기본 스킬은 PII 유입을 이유로 등급 강등하지 않는다(`[v6]`)** — 기본=구동 필수라 강등 시 배포본에서 필수 스킬이 사라져 가용성 회귀. 대신 케이스를 *일반화*(우선)하거나 *cases.local.jsonl로 격리*(차선). 스킬 본체는 기본 유지.
2. **지식 = 본문 개정 메커니즘(케이스 없음) + 탐지 레이어(`[v6]`).** 스킬 본문개정과 **공통 프리미티브**(버전드·자원락·CAS·스냅샷/롤백) 재사용. **단 버전드/롤백은 *되감기*일 뿐 *틀림 탐지*가 아니다** — 케이스를 없앤 만큼 그 안전기능을 본문 안으로 흡수해야 한다:
   - **claim 단위 구조화:** 본문을 문단(claim) 단위로, 각 claim에 인라인 메타 `status(provisional|active|disputed)`·`evidence_level`·`last_verified`·`scope`. 신규 사실 = `provisional` 진입 → 독립 재확인 1회 또는 반박 0회 N경과 시 `active`(스킬 수렴규칙의 사실판).
   - **반박 깃발 채널(최대 결손):** 지식을 *참고한* 에이전트의 "이 사실이 현실과 안 맞았다" 보고 + Reflector 사후판단 → claim을 `disputed`로(본문 불변, 깃발만 — harmful 패턴). *현 설계엔 "지식이 틀렸다"는 입구가 없었음.*
   - **2섹션:** `## 범용 사실` + `## 조건부 사실`(scope·valid_until). law의 "팀 배포는 화요일" 같은 맥락의존 사실을 담는다(P3가 스킬 본문에 금지한 1회성/특정값의 지식판 자리).
   - **모순 게이트:** 개정 후보가 기존 claim과 모순 → 자동 덮어쓰기(last-write-wins) 금지, `conflict` 마킹 후 판단/대표 에스컬레이션. supersede로 근거 연결 보존.
   - **substrate + 라우팅:** 레슨 버퍼(§1)에서 `routing_kind=factual|preference`로 판단된 항목이 지식 개정 후보로. **오분류 복구:** 지식↔스킬 경계 재라우팅 트랜잭션(origin 보존).
   - **must 금지(law §6 정합):** 지식 항목 최고 강제력 = `should_consider`("읽음 확인"까지, "준수" 아님). 절차적 강제가 필요하면 반드시 스킬로 라우팅. 발행게이트(§10-6)는 "지식은 must 대상 아님" 명문화.
3. **등급 재분류 안전(채택):** `skill_id` 안정식별자 + 참조를 id로 + 재분류 트랜잭션 = **원자적 폴더 이동(임시디렉토리 경유 rename)** + cases 동반 + 발견기 인덱스 갱신. (git 추적전환은 git 사용 시 *선택적 부록* — 현재 repo 아님, 배포 경계는 폴더구조로 성립.) 특히 대외비↔기본은 *배포 노출 변화*라 대표 확인.
4. **draft 폐기(채택):** status 라벨 없음. 신규도 실 스킬, 성숙도는 `update_count`/`stats`로 드러냄. (찌꺼기 누적 우려 → 중복 방지는 §4 비차단 점검 + 역방향 중복 플래그가 담당.)
