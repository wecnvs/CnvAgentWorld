# P0 — 확정 스키마·관계 잠금 (구현 계약)

> v0.6 설계의 스키마를 **기존 `LessonLedger.v1`과 어휘 호환**되게 정확히 확정한다.
> P1~ 코드는 이 문서를 계약으로 구현한다. 변경 시 이 문서를 먼저 고친다.
> 근거: `시스템/core/lesson_ledger.py` `_lesson_from_candidate`(:873), `build_lesson_pack`(:746), `_target_resource_path`(:473).

## 1. 레슨 ↔ 케이스 관계 (확정)
- **레슨(`LessonLedger.v1`, `공간/{space}/learning/lessons.jsonl`) = 공간 내부·검증 전·휘발성 버퍼.** 자동 캡처(평가)·growth_gaps·주입은 유지.
- **케이스(`CaseLedger.v1`, `스킬/{등급}/{이름}/cases.jsonl`) = 전역 스킬의 영구 델타.**
- **다리:** 판단 주체가 레슨/평가를 읽고 `routing_kind=procedural`로 판단한 것 → 케이스로 졸업(레슨은 `graduated`). `factual|preference` → 지식 claim으로.
- **스키마 어휘 통일:** 케이스는 레슨의 필드(`applies_when`·`evidence`·`evidence_level`·`confidence`·`status`·`supersedes`·`conflicts_with`·`*_at_utc`·`must_apply`/`application_level`)를 **그대로 계승** + 케이스 전용 필드 추가. → `build_lesson_pack`의 우선순위/3단/예산 로직 재사용 가능(단 `_keyword_matches` 게이트는 미사용).

## 2. CaseLedger.v1 (cases.jsonl 한 줄)
레슨 계승 필드 + 신규 필드. **필수**=등록 계약(없으면 거부).
```jsonc
{
  // --- 식별 ---
  "schema": "CaseLedger.v1",
  "case_id": "stable_id(skill_id, condition, instruction, polarity)",
  "skill_id": "스킬 안정식별자(경로/등급 무관 불변)",   // 필수. 경로 필드는 두지 않음

  // --- 의미 내용 ---
  "condition": "어떤 상황일 때인지(자연어, 키워드 게이트 없음)",
  "instruction": "그 상황엔 이렇게(역할/패턴 일반화, 구체 PII 금지)",  // 필수
  "polarity": "worked | failed",                          // 필수 [신규]
  "applies_when": { "space_id":"", "task_types":[], "agent_modes":[], "resource_paths":[], "keywords":[] },
      // keywords는 저장만(주입 게이트로 안 씀). resource_paths 대신 skill_id 우선.
  "does_not_apply_when": [],
  "conflicts_with": [], "supersedes": [],

  // --- 판단 계약(P1) [신규] ---
  "action": "add_case | supersede | new_skill | none",    // 필수
  "routing_kind": "procedural | factual | preference",    // 필수 (factual/preference면 케이스 아닌 지식으로)
  "judgment_rationale": "왜 이렇게 판단했나",               // 필수
  "source_quote": "근거 발화의 PII 정제 요약(원문 1:1 아님)", // 필수. raw 원문은 cases.local.jsonl

  // --- 보안 ---
  "sensitivity": "public | confidential",                 // 미판단/불확실 = confidential(보수적 기본값)

  // --- 출처/권한 [신규] ---
  "proposed_by": "발의자(채팅/작업/공간관리/관리자/대표)",
  "approved_by": "승인자(대표=자동 / 관리자 / null=미승인)",
  "origin_space": "어느 공간발 판단인지",
  "scope": "global | space",                              // space면 origin_space에서만 주입

  // --- 상태/수렴(레슨 계승 + 확장) ---
  "status": "candidate | provisional_must | active | superseded | retired | conflict | graduated | expired",
  "application_level": "must_apply | may_apply",
  "must_apply": false, "enforcement": "",
  "evidence_level": "user_directive | verified_result | reviewer_approval | agent_observation",
  "confidence": 0.5,
  "evidence": [ { "type":"", "source_message_id":"", "source_event_seq":null,
                  "source_task_id":"", "outcome":"", "source_quote":"(정제)" } ],

  // --- 시각(레슨 계승) ---
  "review_due_at_utc": "",   // provisional_must는 의무
  "valid_until_utc": "", "stale_after_at_utc": "",
  "last_used_at_utc": "", "last_verified_at_utc": "", "created_at": "iso"
}
```
**불변식:** `기본` 등급 스킬의 cases.jsonl에는 `sensitivity:confidential` 케이스를 append 못 함(→ cases.local.jsonl로 강제 라우팅).

## 3. case_events.jsonl (가변 카운터 분리, gitignore)
```jsonc
{ "schema":"CaseEvent.v1", "case_id":"", "skill_id":"",
  "event":"applied | worked | harmful | flagged | review_due | expired",
  "by":"보고 주체", "rationale":"", "at_utc":"iso" }
```
- helpful/harmful 등 **카운터는 frontmatter에 두지 않고 여기서 집계.**

## 4. 파생 성숙도 지표 (저장 안 함, 발견/선택 시 계산)
- `cases` = cases.jsonl active 줄 수 (`cases==0` → "신규(미검증)")
- `worked_ratio` = worked / max(cases,1)
- `harmful` = case_events의 harmful 합 (>0이면 경고, 절대값)
- `stability` = 마지막 본문개정 이후 worked 무사고 횟수 / 최근 개정 간격
- **합산 단일수치(update_count) 금지.**

## 5. SKILL.md frontmatter (불변/저빈도만, 배포 diff·롤백 오염 방지)
```yaml
skill_id: 안정식별자        # 모든 참조의 키. 등급 재분류해도 불변
name: ...
description: ...           # 발견용, 계속 갱신 가능(D3)
version: 1                 # 본문 개정 시만 +1 (CAS 대상)
last_updated: iso          # 본문 개정 시각
```
- 본문에 `## non_overridable`(케이스/개정으로도 못 지우는 안전지침) 섹션 허용.

## 6. 지식 — Knowledge claim (케이스 없음, 본문 내 구조화)
`지식/{등급}/{이름}/지식.md` 본문을 `## 범용 사실` + `## 조건부 사실` 2섹션으로. 각 claim에 인라인 메타:
```yaml
# claim 단위 메타(본문 내 표기)
status: provisional | active | disputed
evidence_level: user_directive | verified_result | reviewer_approval | agent_observation
scope: 자연어 조건 (조건부 사실만; space_id 가능)
valid_until: iso (선택)
last_verified: iso
source_quote: PII 정제 요약
```
- 지식 frontmatter: `knowledge_id, name, description, version, last_updated`(스킬과 동형, cases/non_overridable 없음).
- **지식 최고 강제력 = `should_consider`**(읽음 확인까지; must_apply 금지 — law §6).

## 8. 커밋 게이트 5종 (호출자 무관, 항상 통과) — P1~에서 강제
1. **판단 계약:** `polarity·action·routing_kind·judgment_rationale·source_quote` 충족(없으면 거부).
2. **민감도:** instruction·source_quote·rationale 일반화 + sensitivity 판정(미판단=confidential).
3. **자원 락:** `_with_resource_lock(skill_path)` 안에서만 쓰기.
4. **버전:** 본문 개정은 `expected_version` CAS(불일치 거부).
5. **non_overridable:** 본문 개정 diff가 해당 섹션 약화 시 거부.
- identity는 **자동승인 여부만** 결정(대표=즉시 provisional_must/즉시 적용, 그 외=candidate 수렴).

## 9. skill_id 레지스트리
- `skill_id ↔ 현재 (등급, 경로)` 매핑의 **단일 진실원 = 발견기 인덱스**. 모든 참조는 skill_id로 저장·해석. 등급 재분류 시 인덱스만 갱신(케이스는 폴더 동거라 함께 이동).
