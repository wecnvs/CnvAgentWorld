# 작업계획 승인 — 계획 먼저, 위험도 게이트, 그다음 실행 (구현 계약)

> 채팅에이전트가 작업을 받으면 **"이렇게 하겠다"를 먼저 빠르게 공개하고**, 위험도에 따라
> (낮으면 공간관리 자동승인 / 높으면 대표 승인) 게이트를 통과한 뒤 **실제 작업을 실행**한다.
> P단계 코드는 이 문서를 계약으로 구현한다. 변경 시 이 문서를 먼저 고친다.
> 근거: `시스템/core/room_manager.py` `_handle_chat_agent_result`(:944, 현재 `engine.work()` 직행 :990),
> `시스템/core/chat_result.py` `work_request`(:55), `시스템/core/release_queue.py`(승인 게이트 패턴),
> `시스템/core/engine.py` `work`(:481), `시스템/core/room_manager.py` `_should_auto_continue`(자동연속).

## 0. 문제 (현재 상태)
- `request_work`가 오면 `_handle_chat_agent_result`가 **그 자리에서 `engine.work()`를 동기 실행**한다(:990).
- → 계획 단계 없음, 승인 게이트 없음, 계획/실행 모델 분리 없음. 에이전트가 곧장 전체 작업에 들어간다.
- 목표: 이 한 지점을 **2단계(계획 → 실행)** 로 쪼개고, 그 사이에 **위험도 게이트**를 둔다.

## 1. 전체 흐름 (계약)
```
대표 요청 → 공간관리 pass → 에이전트 깨어남
  [Stage 1: 계획턴]  seat runtime(작은/빠른 모델 허용)
    에이전트 → ChatAgentResult.v1 { action:request_work,
        public_reply:"이렇게 할게요"(방 공개),
        work_request:{ objective, plan:[...], needs_approval, approval_reason, risk } }
    시스템: engine.work() 호출 안 함 → work_plan.register() = pending_approval
  [게이트]  needs_approval = (에이전트 명시 선언) OR (시스템 휴리스틱 high)
    false → 공간관리 자동승인(다음 auto_continue tick)
    true  → 방에 '결재 말풍선' 발행 + 하이라이트 → 대표가 말풍선에서 [진행]/[반려]
  [Stage 2: 실행턴]  work runtime(큰 모델) — 승인 후에만
    engine.work() 실행 → 기존 heartbeat/steering/release 흐름
  완료 → 결과 release → 방 공개
```

## 2. WorkPlan.v1 (작업계획 원장 한 줄)
저장 위치: `공간/{space}/work_plans.jsonl` (append-only, release_queue와 동형). 최신 상태는 `plan_id`별 마지막 줄.
```jsonc
{
  "schema": "WorkPlan.v1",
  "plan_id": "stable_id(space, requesting_agent, worker, objective, intent_id)",  // 필수, 멱등키
  "space_id": "",
  "requesting_agent": "계획을 낸 채팅에이전트 토큰",   // 필수
  "worker": "실제 작업할 에이전트 토큰",               // 필수 (자기 자신일 수 있음)
  "objective": "무엇을",                              // 필수
  "plan_steps": ["1. ...", "2. ...", "3. ..."],       // 필수, 1개 이상
  "constraints": [],
  "needs_approval": true,                             // 필수, 게이트 결과. = 에이전트선언 OR 시스템high
  "approval_reason": "지침 변경 포함 / 대외비 공유 등",  // 승인 말풍선에 표시할 사유
  "risk": {
    "agent_needs_approval": true,                    // 에이전트 명시적 선언 (없으면 true 보수 처리)
    "agent_level": "low | high",                     // 에이전트 자가평가(보조)
    "agent_reason": "",
    "system_level": "low | high",                    // 시스템 휴리스틱 산출(격상 전용)
    "system_signals": ["guide_change", "confidential_share", "external_publish", ...]
  },
  "state": "pending_approval | approved | rejected | superseded | executing | done | error",  // 필수
  "approval_mode": "auto_manager | representative",   // needs_approval=false → auto_manager, true → representative
  "approval_message_id": "",                          // 방에 올린 '결재 말풍선'의 message_id (하이라이트 대상)
  "approved_by": "", "approved_at_utc": "",
  "rejected_by": "", "reject_reason": "",
  "task_id": "",                                      // Stage 2에서 생성된 작업코드
  // --- 오케스트레이션 결속 (effect/generation 펜스 호환) ---
  "intent_id": "", "conversation_thread_id": "", "room_generation": 0,
  "source_event_seq": 0, "source_message_id": "",
  "manager_claim_token": "", "manager_fencing_token": "",
  "created_at_utc": ""
}
```

## 3. ChatAgentResult.v1 확장 (하위호환)
`chat_result.work_request`(:55)가 반환하는 dict에 필드 추가. **plan/risk가 없으면 기존 동작 유지**(plan_steps=[objective] 단일 단계, needs_approval=true 보수 기본).
```jsonc
"work_request": {
  "objective": "...",            // 기존
  "suggested_worker": "...",     // 기존
  "constraints": [...],          // 기존
  "plan": ["1. ...", "2. ..."],  // [신규] 단계 목록
  "needs_approval": true,        // [신규] 에이전트가 "이건 승인 필요"를 명시적으로 선언 (핵심)
  "approval_reason": "지침 변경을 포함함",  // [신규] 왜 승인이 필요한지(승인 말풍선에 표시)
  "risk": { "level": "low|high", "reason": "..." }  // [신규] 자가평가(보조)
}
```
- **에이전트의 명시적 선언이 1차 게이트다.** `needs_approval=true`면 위험도와 무관하게 대표 승인 말풍선을 올린다.
- `work_request()`에 `plan`(list[str], 정제·상한 12), `needs_approval`(bool), `approval_reason`(str), `risk`(dict→{agent_level,agent_reason}) 파싱 추가.
- 누락 시 기본값: `plan=[objective]`, `needs_approval` 키가 없으면 **None(미선언)** 으로 둔다.
- **미선언(None)은 시스템 위험도로 판정한다(위험도 기반 혼합).** 평범한 작업은 자동 진행(무회귀),
  지침변경·대외비·외부발행·대량삭제·고비용 신호가 있으면 시스템이 승인행으로 격상한다.
  (이전 보수안 "미선언→무조건 승인"은 폐기 — 대표가 고른 '혼합 위험도 기반'과 어긋나고 모든 작업이
  승인 대기로 급변하기 때문. 에이전트 플래그 + 시스템이 위험한 것만 잡는 모델로 확정.)

## 4. 승인 필요 판정 (명시적 선언 + 시스템 백스톱)
`work_plan.assess_approval(objective, plan_steps, agent_needs_approval, agent_risk_level)`
→ `(needs_approval, system_level, signals)`.

**게이트 = `needs_approval = (에이전트 명시적 선언 True) OR (시스템 휴리스틱 high)`.**
- 즉 **에이전트가 "승인 필요"라고 하면 무조건 승인행**(에이전트의 판단 존중 + §0 거짓신고 방지).
- 에이전트가 "불필요"·미선언이어도 **시스템이 high 신호를 잡으면 승인행으로 격상**(백스톱). 시스템은 절대 *낮추지* 않는다 — 격상만 한다.
- 둘 다 아닐 때만 자동 진행(`auto_manager`). → 미선언+무신호 = 자동(무회귀).

시스템 휴리스틱 `system_level=high` 신호 (objective/plan/constraints 스캔):
  - **지침·규칙 변경**: `law*.md`, `*지침*`, `CLAUDE.md`/`AGENTS.md`, role.md, 공간지침 등 규범 파일 수정·생성·삭제 ← **반드시 승인**
  - **대외비 공유/반출**: `자산/대외비`·`*/대외비/*`·`sensitivity=confidential` 자산을 읽어 외부/타 공간/방에 공유·게시·전송 ← **반드시 승인**
  - **외부 발행/전송**: 메일·게시·배포·푸시·공개 URL·결제 등
  - **대량/비가역 파일 변경**: 삭제·덮어쓰기·이동·다수 파일, 자기 작업 자리 밖 경로 접근
  - **고비용**: 단계 수 과다·외부 API 반복·장시간 추정
  - 그 외 → `system_level=low`
- `risk.agent_level`은 보조 신호로 기록(`needs_approval`이 1차 게이트). 키워드 목록은 코드 상수로 시작, 향후 `지식/`의 위험기준 자원과 발견기 연동 가능(범위 밖).
- **불확실하면 승인행**(보수적 기본). 판정 불가/예외 발생 시 `needs_approval=true`로 폴백.

## 5. 게이트 흐름
### 5.1 `_handle_chat_agent_result` 재배선 (room_manager.py:944)
- `engine.work()` 직행 제거 → 다음으로 교체:
  1. `request = chat_result.work_request(...)` (plan/risk 포함)
  2. `risk = work_plan.assess_risk(objective, plan, agent_level)`
  3. `plan = work_plan.register(space, ..., state="pending_approval", approval_mode=...)`
  4. `public_reply`("이렇게 할게요")는 방에 공개(기존 announce_only 경로 재사용).
  5. 응답의무 = **assigned**(작업계획 대기). 아직 task 없으니 `delegate_to_task` 아님 → `assign_for_context`(parallel_pass가 쓰는 것과 동형).
  6. effect 기록(`work_plan_registered`), 멱등은 `plan_id`/effect_id로.
- 반환: 계획만 등록됐음을 알리는 라우팅 문자열(작업 미실행).

### 5.2 needs_approval=false → 공간관리 자동승인
- `_should_auto_continue`(이미 패치됨)에 **"pending_approval & approval_mode=auto_manager 인 plan이 있으면 연속"** 조건 추가 → `_has_pending_candidates`와 같은 결의 `_has_auto_approvable_plan(space)`.
- 다음 auto_continue tick에서 공간관리 스냅샷에 plan이 보이고, 시스템이 자동 승인 → `work_plan.approve(auto)` → `engine.work()` 실행 → 결과 release.
- 별도 대표 개입 없음. (방금 고친 자동연속이 여기서 트리거가 된다.)

### 5.3 needs_approval=true → 대표 승인 (대화창 결재 말풍선)
- plan 등록 시 **방에 '결재 말풍선'을 1개 발행한다**(공개 publish 경로). 내용: 계획 요약 + `approval_reason`(왜 승인 필요한지). 이 말풍선의 `message_id`를 `plan.approval_message_id`에 저장.
- 시스템이 `approval_required` 마커를 쓴다(아래 5.5). 대시보드는 이 말풍선을 **하이라이트 + [진행]/[반려] 버튼을 말풍선 안에** 렌더한다(기존 `handback-highlight` 패턴 확장).
- 승인/반려는 **대화창 말풍선에서 직접** 누른다 → 라우터 `POST /api/spaces/{space}/plans/{plan_id}/approve|reject` (release_queue approve 패턴 복제).
- 대표 승인 → `work_plan.approve(representative)` → `engine.work()` 실행.
- 응답의무는 승인 전까지 assigned 유지. 대표가 새 발언하면 generation 펜스로 stale plan 차단(기존 메커니즘).

### 5.5 결재는 무조건 대화창 말풍선에서 (UX 불변식 — 실수 금지)
- **불변식 A:** `needs_approval=true`인 plan은 **반드시** 방에 결재 말풍선(`approval_message_id`)을 남긴다. 말풍선 발행에 실패하면 plan을 `pending_approval`로 두고 실행하지 않는다(자동실행 절대 금지).
- **불변식 B:** `needs_approval=true`인 plan은 `auto_manager`로 절대 승인되지 않는다(코드에서 assert). 자동승인 경로는 `needs_approval=false`만 통과.
- **불변식 C:** 승인 전까지 `engine.work()`는 호출되지 않는다. (계획턴과 실행턴 사이 게이트가 유일한 진입점.)
- 마커: `공간/{space}/approval_required.json`(복수면 jsonl). 필드: `plan_id`, `highlight_message_id`(=approval_message_id), `approval_reason`, `worker`, `at`. 대표 발언/승인/반려 시 해제.
- 프런트: `/api/spaces/{space}/approvals` 폴링(또는 status에 포함) → 해당 message_id 말풍선에 `approval-highlight` 클래스 + "결재 필요" 배지 + [진행]/[반려] 버튼. (기존 `room-chat.js:365,369` handback 렌더 분기 재사용.)

### 5.6 실행(Stage 2)
- 승인 처리 함수가 `engine.work(worker, space, objective+constraints, context, requested_by=plan_id, approved_by=approval_mode)` 호출.
- plan_steps를 `engine.work`의 지시/결과.md 초기 체크리스트로 주입(기존 "작업 분할·체크포인트" 프롬프트와 결합).
- 작업 모델: seat 기본과 다르게 두려면 `runtime.write_runtime(wdir, ...)` 오버라이드(이미 engine.work가 지원, :502).
- 상태 전이: plan.state `approved→executing→done|error`, 응답의무 assigned→delegated→closed.

## 6. 상태/전이 요약
| 단계 | plan.state | response_obligation | 모델 |
|---|---|---|---|
| 계획 공개 | pending_approval | assigned | seat(작은 모델 허용) |
| 자동승인(needs_approval=false) | approved→executing | delegated | work(큰 모델) |
| 결재 말풍선 대기(needs_approval=true) | pending_approval | assigned | — |
| 대표 승인 | approved→executing | delegated | work(큰 모델) |
| 완료 | done | closed | — |
| 반려 | rejected | closed(rejected) | — |

## 7. 재사용 자산 매핑 (새 코드 최소화)
- 승인 게이트(claim+approve/reject) → `release_queue.py` 패턴 복제 → `core/work_plan.py`.
- 승인 후 작업 실행 → `engine.work()` 그대로.
- 상태추적 → `response_obligation`(assign/delegate/close 기존 API).
- low 자동승인 트리거 → `_should_auto_continue`(이미 패치) 확장.
- 모델 분리 → seat `agent_runtime.json` vs work `runtime.write_runtime`(기존).
- 공간관리 가시성 → `_prompt_room_status_snapshot`에 `work_plans` 섹션 추가(candidate_queue 섹션과 동형, room_manager.py:~3130).

## 9. 구현 순서 (P단계)
- **P1.** ✅ `core/work_plan.py`: 원장(register/get/list/approve/reject/mark_executing/mark_finished/supersede) + `assess_approval`. 단위 테스트 20.
- **P2.** ✅ `chat_result.work_request`: plan/needs_approval/risk 파싱 + 하위호환. 단위 테스트 8.
- **P3.** ✅ `_handle_chat_agent_result` 재배선(계획 등록 + 게이트 + 저위험 인라인 즉시 실행 / 고위험 결재 대기). 통합 테스트.
- **P4.** ⊘ 불필요 — 저위험을 P3에서 인라인 자동승인+즉시 실행(`_execute_work_plan`)으로 처리해 자동승인 대기 plan이 큐에 남지 않음. (auto_continue 트리거 별도 불필요. 단, parallel_pass 후보용 auto_continue 패치는 별건으로 유지.)
- **P5.** ✅ 결재 라우터 `/approvals`·`/plans/{id}/approve|reject` + `_run_agent_turn`이 결재 말풍선 message_id를 plan에 연결 + 대화창 [진행]/[반려] 버튼(handback-highlight 확장). 라우터 테스트.
- **P6.** ⬜ law_chat.md/law_space.md 반영(미착수).

## 10. 하위호환·안전
- plan/선언 없는 기존 `request_work` → `plan=[objective]`, `needs_approval=None(미선언)` → **시스템 위험도로 판정**. 평범하면 자동 실행(기존 동작 보존·무회귀), 지침변경·대외비·외부발행·대량삭제·고비용이면 결재 대기. (P3 구현: 저위험은 `_handle_chat_agent_result`가 즉시 `engine.work` 실행 → 종전과 동일 UX.)
- 멱등: `plan_id` = stable_id, effect_id로 중복 등록·중복 승인 차단(release_queue와 동일 원리).
- generation/claim 펜스: plan 승인·실행도 `manager_claim.is_current` + `orchestration.is_context_stale` 검사로 stale 차단.
- 무한루프: low 자동승인도 `AUTO_CONTINUE_MAX_TURNS` 상한 안에서만 연속.
