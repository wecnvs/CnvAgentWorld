# Tool Creator

`도구/`에 **범용·재사용 가능한 원자 기능**을 만든다. 도구는 부품, 스킬은 그 부품의 조립품이다(law.md §7).

## 무엇이 '도구'인가 — 판단
판단 기준 단 하나: **"여러 스킬·작업이 또 쓸 만한가?"**
- Yes → 도구로 만든다(`도구/`).
- 이 스킬에만 특수 → 도구가 아니다. 그 스킬의 `scripts/`에 둔다.

곱셈·CSV파싱·HTTP요청·재시도 래퍼 = 도구. "이 회사 전용 결재 규칙 계산" = 도구 아님(특정 스킬 로직).

## 만들기 절차
1. **범용성 판단**(위). 범용이면 진행.
2. **등급 선택**: `도구/{기본|추가|고급|대외비}/<tool-name>/`
   - 기본=구동 필수 / 추가=일반 / 고급=특수 / 대외비=민감.
   - 도구 코드엔 **비밀을 넣지 않는다**(범용 코드만). 비밀이 필요하면 `자산/대외비/`에 두고 런타임에 환경변수·상대경로로 읽는다.
3. **작성**:
   ```
   도구/<등급>/<tool-name>/
   ├── 도구.md     # frontmatter: name, description, entry, runtime
   └── <코드>       # 단일 책임, 작게
   ```
   `도구.md` frontmatter:
   ```yaml
   ---
   name: <tool-name>
   description: <무엇을 + 언제 쓰는지 + 사용자 표현 예시 + 핵심 용어>
   entry: <진입 파일[:함수]>     # 예: multiply.py:multiply
   runtime: <python|node|sh ...>
   ---
   ```
4. **DoD대로 실제 실행 검증**: 코드를 돌려 의도한 출력이 나오는지 확인(엣지 포함).

## 발견 잘 되게 — `description`이 전부 (태그 없음)
도구는 별도 등록·태그 없이 **frontmatter `description`으로 발견기에 잡힌다.** 그러니 description을 *무엇을 + 언제 쓰는지 + 사용자가 그걸 말할 법한 표현 + 핵심 용어*로 또렷이 쓴다. 모호하면 영영 안 찾힌다. `entry`·`runtime`도 채워, 찾은 즉시 호출 가능하게.

## 작게·단일 책임
한 도구 = 한 일. 커지면 쪼갠다. 특정 환경에서 실패하면 코드를 새로 짜지 말고 조건/fallback을 더해 어디서든 돌게 보강한다.

## 다른 문서가 이 도구를 쓰는 법 (law.md §8)
스킬·지식은 `관련 도구: 두 수 곱하기 — 발견기로 찾아 사용`처럼 **자연어로** 적고, 사용 시점에 `발견기`로 찾는다.

## 완료 기준 (DoD)
- [ ] `도구/<등급>/<이름>/도구.md`(name·description·entry·runtime) + 코드 실제 생성.
- [ ] `description`이 발견 최적화 형식(무엇·언제·표현·용어)으로 또렷함.
- [ ] `발견기`로 이 도구가 실제로 후보에 뜨는지 확인(`python3 도구/기본/발견기/발견.py "<관련 표현>" --type tool`).
- [ ] 코드가 실제로 실행되어 의도한 출력을 냄.

## 관련 스킬
- `skill-creator` — 스킬을 만들다 범용 기능이 필요하면, 인라인으로 박지 말고 이 도구-크리에이터로 먼저 만든다.
- `발견기` — 만든 도구가 잘 찾히는지 확인.
