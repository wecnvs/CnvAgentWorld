# 발견기 (스킬)

요청에 맞는 자원을 **frontmatter description 기반으로** 찾아 **후보 여러 개**를 추리고, 그중 골라 활용한다. (태그 없음 — 설명으로 찾는다.)

## 언제
- 요청을 받아 무슨 스킬·지식이 도움될지 찾을 때 (law.md §6의 3단계).
- 작업 중 도구·자산·지식이 필요한데 뭐가 있는지 모를 때 (스스로 발견).
- 단순 대화·질의응답엔 쓰지 않는다(트리아지 — 바로 답).

## 어떻게
1. **발견 도구 실행:**
   ```
   python3 도구/기본/발견기/발견.py "<찾는 내용>" [--type skill|knowledge|tool|asset|all] [--top 5]
   ```
   → 점수순 후보 목록(이름·종류·설명·경로[, 도구는 호출법])을 받는다.
2. **후보군에서 고른다.** 하나로 단정하지 말고, 후보들의 `description`을 비교해 지금 요청에 맞는 **1~몇 개**를 고른다. 여럿을 조합해도 된다.
3. **활용** (law.md §6-5):
   - 지식 = 참고 / 스킬 = 사용(그대로 복제 말고 적재적소 변형) / 도구 = `entry`·`runtime`대로 호출 / 자산 = 참조.
4. **없으면** 다른 표현으로 재검색 → 그래도 없으면 발견기 없이 직접 판단.

## 왜 후보를 여러 개 주나
정답 하나를 단정하지 않는다. 에이전트가 후보를 비교·조합해 맥락에 맞게 고르는 게 더 똑똑하고, 한 번 헛짚어도 차선이 보인다.

## 규모가 커지면 (다단계)
카탈로그가 크면 한 번에 다 보지 말고: ① 넓은 키워드로 1차 후보 → ② 좁은 질의로 재검색 → ③ 최종 후보. 발견 도구를 좁혀가며 여러 번 호출해 단계적으로 추린다. (컨텍스트를 아끼는 핵심.)

## 잘 찾히게 하려면 (만드는 쪽 책임)
발견 품질은 결국 **자원의 description 품질**이다. 자원을 만들 땐 frontmatter `description`에 *무엇을 + 언제 쓰는지 + 사용자 표현 예시 + 핵심 용어*를 또렷이 담는다. (각 구축 스킬 — skill/tool/knowledge/asset-creator — 이 이를 강제한다.)
