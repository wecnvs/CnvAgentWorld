# Knowledge Creator

`지식/`에 **참고용 내용**을 만든다. 스킬이 '하는 법'이면, 지식은 '참고할 내용'이다. 지식은 **사용이 아니라 참고**된다(law.md §6).

## 무엇이 '지식'인가
판단·작업에 근거가 되는 정리·기준·맥락·자료. 절차를 실행하는 것이면 그건 지식이 아니라 스킬이다.

## 만들기 절차
1. **등급 선택**: `지식/{기본|추가|고급|대외비}/<knowledge-name>/`
   > **🔒 대외비 판정 (보안 — law.md §7):** 개인정보(이름·연락처·계정)·특정 회사/고객/기관 정보·내부 전용 사실이 들어가면 **반드시 `대외비/`**. 공개 깃에는 `기본/`만 올라가므로(추가/고급/대외비 제외), 민감 사실이 `기본/`에 들어가면 공개 유출이다. 일반화 못 하면 대외비, 애매하면 더 안전한 등급으로.
2. **작성**:
   ```
   지식/<등급>/<knowledge-name>/
   ├── 지식.md     # frontmatter(name, description) + 본문
   └── <첨부>       # 선택
   ```
   `지식.md` frontmatter:
   ```yaml
   ---
   name: <knowledge-name>
   description: <무엇에 대한 참고 지식인가 + 언제 참고하나 + 사용자 표현 예시 + 핵심 용어>
   ---
   ```

## 발견 잘 되게 — `description`이 전부 (태그 없음)
지식은 별도 등록·태그 없이 **frontmatter `description`으로 발견기에 잡힌다.** 발견기는 본문 전체를 읽어 의미 추론하는 것이 아니라, 매니페스트 frontmatter의 `description`을 스캔해 후보를 만든다. 따라서 description 작성이 지식 생성의 핵심이다.

### `description` 작성 규칙 (필수)
- **무엇**: 이 지식이 무엇에 대한 참고자료인지 쓴다.
- **언제**: 어떤 요청·작업·판단 상황에서 참고해야 하는지 쓴다.
- **사용자 표현 예시**: 사용자가 실제로 말할 법한 표현을 2~4개 넣는다. 예: `'배포 규칙 알려줘'`, `'회의 결정사항 찾아줘'`, `'API 키 관리 기준 참고'`.
- **핵심 용어**: 검색에 걸릴 명사·동의어·약어를 넣는다. 예: `배포, 릴리즈, 운영, 승인, 롤백`.
- **한 줄 유지**: 현재 발견기는 단순 frontmatter 파서이므로 `description:` 값은 줄바꿈 목록이 아니라 한 줄 문장으로 쓴다.
- **태그 금지**: 태그·수동 인덱스를 만들지 않는다. 필요한 검색어는 description 문장 안에 자연스럽게 넣는다.

좋은 예:
```yaml
description: 배포와 릴리즈 운영 기준 참고 지식. 배포 승인·롤백·점검창 판단이 필요할 때 참고한다. '배포 규칙 알려줘', '릴리즈 전에 뭐 확인해', '롤백 기준 찾아줘'. 핵심 용어: 배포, 릴리즈, 운영, 승인, 롤백, 점검.
```

나쁜 예:
```yaml
description: 배포 문서
```

본문은 단독으로 읽혀도 이해되게 쓰되, 발견 품질은 description에서 먼저 결정된다고 보고 작성한다.

## 본문에서 도구·자산·다른 지식 참조 (law.md §8)
특정 문서를 박지 말고 **자연어로** 적고 발견기로 찾는다: `관련 도구: …`, `참고 지식: …`.

## 완료 기준 (DoD)
- [ ] `지식/<등급>/<이름>/지식.md`(name·description + 본문) 실제 생성.
- [ ] `description`이 발견 최적화 형식(무엇·언제·사용자 표현 예시·핵심 용어)으로 또렷함.
- [ ] `발견기`로 이 지식이 후보에 뜨는지 최소 2개 표현으로 확인:
      `python3 도구/기본/발견기/발견.py "<사용자 표현>" --type knowledge`
      `python3 도구/기본/발견기/발견.py "<핵심 용어 조합>" --type knowledge`
