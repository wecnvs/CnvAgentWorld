# 앱등록 — 워크스페이스 앱을 만들고/가져와 대시보드 앱 탭에 올린다

## 언제 쓰나
대표가 **"앱 만들자 / 앱 등록하자 / 앱 개발하자 / 이 앱 가져오자 / 블렌더(외부 프로그램) 추가하자 / 웹앱 올려줘 / 레빗 애드인 등록"** 같이 말하면 이 스킬을 쓴다.
결과물은 항상 같다 — `앱/<등급>/<이름>/` 폴더 + `앱.md` 매니페스트를 만들어 **대시보드 뷰어 → 앱 탭에 그 앱이 종류에 맞는 버튼과 함께 뜨게** 한다.

루트폴더의 `앱/안내.md`가 규약의 정본이다. 이 스킬은 그 규약대로 앱을 만드는 *방법*이다.

## 앱이란 / 어디에 사는가
- 한 앱 = `앱/<등급>/<이름>/` 한 폴더. 그 안에 **소스코드·설치파일·실행기**가 함께 있다.
- 대시보드 앱 탭이 `앱/`을 라이브 스캔해 `앱.md` frontmatter로 표시·동작한다(등록·태그 불필요).

## 1단계 — 종류(kind) 판정 (결정표, 추측 금지)
사용자 말과 앱의 성격으로 아래에서 하나를 고른다.

| 이런 앱이면 | kind | 대시보드 동작 |
|----|----|----|
| 우리가 만든 **웹 애플리케이션**(브라우저로 쓰는 것) | `web-app` | 실행 → 서버가 포트 열고 **내 브라우저**가 그 포트를 연다 |
| 우리가 만든 **독립실행형**(서버에서 도는 프로그램) | `standalone` | 실행 → **서버 호스트**에서 실행(에이전트가 MCP로 사용) |
| **외부 프로그램**(예: Blender) — 소스 없음, 설치/실행파일만 | `external` | 실행 → **서버 호스트**에서 실행 |
| **Revit 애드인**처럼 설치해서 쓰는 것(실행파일 없음) | `revit-addin` | 설치파일 **다운로드만** |
| 기타 설치형(다운로드만 필요) | `install-only` | 설치파일 **다운로드만** |

판단 규칙:
- "웹/브라우저/대시보드/포트로 접속" → `web-app`
- "서버에서 돌려/실행돼서 에이전트가 쓰게/렌더링/백그라운드 프로세스" → `standalone`(우리 것) 또는 `external`(블렌더 등 남이 만든 것)
- "Revit 애드인/플러그인/설치해서 써" → `revit-addin`
- 실행할 게 없고 설치파일만 받게 하면 됨 → `install-only`

## 1.5단계 — 실행 위치(target) 판정 (어디서 도는가)
앱은 **서버 컴퓨터뿐 아니라 네트워크 원격 윈도우·VM에서도** 실행할 수 있다. `target`으로 정한다(미지정=`local`=서버). 대시보드가 카드에 실행 위치 배지로 구분하고 그 자리에서 실행·중지한다.

| 이런 경우 | target | 채널 | 비고 |
|----|----|----|----|
| 서버 컴퓨터에서 실행 | (생략) 또는 `local` | local(subprocess, Win/mac/Linux) | 기본 |
| **네트워크 윈도우/VM에서 GUI앱**(예: 원격 윈도우의 Revit) | 별칭(예: `원격윈도우PC`) | `cu-helper` | **권장** — 실행·중지·상태 완전 추적 |
| 이 Mac의 Parallels 게스트 | 별칭(예: `건축VM`) | `parallels` | `prlctl exec` — 추적 제한 |
| OpenSSH 있는 원격 | 별칭 | `ssh` | `ssh` — 추적 제한 |

- **원격/VM 타깃은 레지스트리에 먼저 등록**한다(대외비): `자산/대외비/앱실행대상/targets.json`
  `{"targets":[{"name":"원격윈도우PC","channel":"cu-helper","host":"<host/IP>","port":8599,"label":"원격 윈도우 PC"}]}`. 실제 host/IP·VM명·자격증명은 **대외비**(law.md §7) — 매니페스트·로그·공개경로에 평문 금지. 그다음 앱.md에 `target: 원격윈도우PC`만 적는다.
- **대표 시나리오(원격 윈도우 레빗)**: 그 윈도우에 `cu_helper.ps1`을 대화형 세션에서 띄우고(도구 `vm-cu-helper`), `cu-helper` 타깃으로 등록 → 앱.md `target:`+`run: "C:\…\Revit.exe"`. 앱 탭 '실행'이 그 윈도우에서 레빗을 띄우고 '중지'로 끈다. 띄운 뒤 조작은 `computer-use-win`/`vnc-mcp-원격실행`을 발견기로 찾아 쓴다.
- 헬퍼/VM이 안 떠 있거나 레지스트리에 없으면 실행이 실패/안내된다 — 먼저 타깃을 준비한다.

### target = 실행 위치 **이자** 컴퓨터유즈(원격제어) 위치
앱을 어느 컴퓨터/VM에서 띄울지(`target`)를 정하면, **그 앱을 조작하는 컴퓨터유즈도 그 target에서** 이뤄진다 — 둘은 같은 자리다. 등록 시 이 한 가지(target)로 "어디서 실행 + 어디서 제어"가 함께 정해진다.
- `local`(서버 Mac) → **서버 화면(맥 콘솔)** 에서 조작. 에이전트 자동조작은 `computer-use-mac`.
- `cu-helper`(원격 윈도우/VM, 예: `도윤컴-호스트`·`도윤컴-VM-A`) → 대시보드 **앱 탭 「원격 세션」 카드 또는 앱 카드의 `🖥 원격제어`** 로 그 화면을 브라우저에서 보고 클릭/타이핑한다. 패널에서 **보기 전용 ↔ 제어** 를 토글로 고른다(여러 세션을 「▦ 전체 그리드로 열기」로 동시 보기·조작). 에이전트 자동조작은 `computer-use-win`/cu_helper(`/screenshot·/click·/type·/key`).
- 즉 대표가 "이 앱은 **서버에서/원격 윈도우에서/그 안 VM에서** 실행하고 거기서 제어해" 라고 하면 → 그 자리에 해당하는 `target`(local / cu-helper 별칭)을 골라 등록하면 실행·원격제어가 그 컴퓨터에서 일어난다. 원격/VM 별칭과 연결정보는 대외비 레지스트리(`자산/대외비/앱실행대상/targets.json`)에 둔다.

## 2단계 — 등급 판정 (기본/추가/고급/대외비)
- **기본** — 워크스페이스에 꼭 필요한 *가벼운 공개* 앱만. (Git 추적 — 큰 바이너리·사내용 금지)
- **추가** — 일반 확장. (로컬 전용)
- **고급** — 고급·특수. (로컬 전용)
- **대외비** — 회사/고객·내부 전용, 또는 **큰 바이너리/외부 프로그램**(Blender·사내 레빗 애드인 등). (로컬 전용, Git 제외)

판단 규칙(애매하면 더 안전한 상위 등급):
- 설치파일·실행 바이너리가 무겁다 → **추가 이상**(기본 금지).
- 회사명·고객·내부 자료·자격증명이 얽힌다 → **대외비**(law.md §7 보안 불변식).
- 외부 프로그램(블렌더 등)·사내 애드인 → 보통 **대외비** 또는 **추가**.

## 3단계 — 폴더 scaffold
```bash
# <등급>·<이름>을 1·2단계 판정으로 채운다. 이름은 공백 대신 하이픈.
mkdir -p "앱/<등급>/<이름>/소스" "앱/<등급>/<이름>/설치" "앱/<등급>/<이름>/실행"
```
- `소스/` — 우리가 개발하는 경우 소스코드(외부 프로그램이면 비워둠).
- `설치/` — 배포·설치파일(.msi·.addin·.dmg 등). '설치파일 다운로드' 대상.
- `실행/` — 실행기·실행파일(런처·.app·run.sh 등). '실행' 대상.
- 종류에 필요 없는 폴더는 비워두거나 안 만들어도 된다.

## 4단계 — 파일 배치
- **web-app**: 소스를 `소스/`(또는 앱 폴더)에 둔다. 서버를 띄우는 실행기/명령을 준비한다(포트는 `0.0.0.0` 바인드).
- **standalone**: 실행기를 `실행/`에 둔다(우리 빌드 산출물).
- **external**(블렌더 등): 설치파일을 `설치/`에, 실행기/런처를 `실행/`에(또는 시스템에 설치된 실행경로를 `run`에 직접). 소스는 없음.
- **revit-addin / install-only**: 설치파일을 `설치/`에 둔다.

## 5단계 — `앱.md` 매니페스트 작성 (정본)
앱 폴더 루트에 `앱.md`를 만든다. **종류에 맞는 필드만** 채운다.

공통 frontmatter:
```yaml
---
name: <앱 이름>
description: <무엇을 하는 앱 + 언제 쓰나 + 핵심 용어>   # 앱 탭이 잘 찾게
kind: web-app | standalone | external | revit-addin | install-only
target: <생략=local | 원격/VM 별칭>   # 선택 — 어디서 실행(1.5단계). 미지정=서버
platform: win | mac | cross | web     # 선택
version: 0.1.0                        # 선택
status: ready | wip | deprecated      # 선택
---
# 사용법/설치법/주의 (본문)
```

종류별 추가 필드:
| kind | 채울 필드 | 예 |
|----|----|----|
| web-app | `port`, `run`(그 포트를 0.0.0.0에 여는 서버 시작) | `port: 8090` / `run: python3 -m http.server 8090 --bind 0.0.0.0` |
| standalone | `run`(서버 호스트에서 실행) | `run: 실행/run.sh` 또는 `run: ./MyApp` |
| external | `run`(서버에 설치된 프로그램 실행), 필요시 `install` | `run: open -a Blender` / `install: 설치/blender.dmg` |
| revit-addin | `install`(설치파일 경로) | `install: 설치/MyAddin-Setup.msi` |
| install-only | `install` | `install: 설치/Setup.exe` |

> **web-app 핵심**: 서버는 반드시 `0.0.0.0`에 바인드한다. `localhost`/`127.0.0.1`로 바인드하면 Tailscale 등 다른 컴퓨터에서 안 열린다. 대시보드는 '실행' 뒤 `http://<내가-접속한-호스트>:<port>`를 내 브라우저로 연다.
> **실행 안전**: 대시보드는 매니페스트의 `run`만 실행한다(클라이언트 임의 명령 차단). `run`은 가급적 앱 폴더 기준 상대경로/명령으로, 절대경로 하드코딩 지양(외부 프로그램 실행경로는 예외).
>
> **`run`은 그 target의 OS에 맞춘다 (대시보드 실행은 Win/mac/Linux 모두 지원):**
> - mac: `open -a Blender`(런처 — 추적 안 됨) 또는 바이너리 직접 `/Applications/X.app/Contents/MacOS/X`(추적/중지 됨)
> - Windows: 실행파일 경로 `"C:\\Program Files\\…\\App.exe"` (따옴표로 공백 처리). web-app은 `python -m http.server <port> --bind 0.0.0.0`
> - 원격/VM(`cu-helper`): `run`은 그 머신 기준 경로(예: 원격 윈도우의 `"C:\\…\\Revit.exe"`). 대시보드가 그 머신에서 띄운다.
> - 런처(`open -a`·`start`)는 즉시 끝나 '실행 중/중지' 추적이 안 된다 → 추적·중지가 필요하면 **바이너리/서버를 직접** 실행하는 명령으로 적는다.

## 자연어 → 입력 변환 (에이전트가 수행, 예시 ≥3)
| 사용자가 한 말 | kind | 등급(기본값) | 핵심 필드 |
|----|----|----|----|
| "내가 만든 견적 웹앱 올려줘, 8091 포트로 띄워" | web-app | 추가 | `port: 8091`, `run: <서버 시작, 0.0.0.0:8091>` |
| "블렌더를 앱에 추가해서 버튼으로 켜지게" | external | 대외비 | `run: open -a Blender` |
| "우리 레빗 애드인 설치파일 받을 수 있게 등록" | revit-addin | 대외비 | `install: 설치/<애드인>.msi` |
| "서버에서 도는 변환기 실행기 등록, 실행되면 에이전트가 쓰게" | standalone | 추가 | `run: 실행/<런처>` |

기본값: 단위/포트 미명시 web-app → 대표에게 포트 1개 묻거나 8090부터 빈 포트. 등급 미명시 → 외부/바이너리는 대외비, 그 외 추가. 이름에 공백 → 하이픈.

## 6단계 — 검증 (DoD — 이게 끝나야 완료)
1. **매니페스트 존재**: `앱/<등급>/<이름>/앱.md`가 실제로 있고 frontmatter에 `name`·`description`·`kind`가 채워져 있다.
2. **앱 탭에 뜬다**: 아래로 목록에 잡히는지 확인한다.
   - 파일 직접: `python3 -c "import sys; sys.path.insert(0,'시스템'); from core import apps; import json; print(json.dumps([a['name']+'|'+a['kind']+'|'+a['grade'] for a in apps.list_apps()['apps']], ensure_ascii=False))"`
   - 또는 대시보드 떠 있으면: `curl -s http://localhost:8686/api/apps`
   내 앱이 목록에 있고 `kind`가 맞는지 본다.
3. **종류별 동작 확인**:
   - web-app → `run`의 포트가 실제로 열리는지(실행 후 `http://<호스트>:<port>` 접속). 서버가 `0.0.0.0` 바인드인지 확인.
   - standalone/external → '실행'이 서버 호스트에서 프로세스를 띄우는지(`/api/apps` 의 `running:true`).
   - revit-addin/install-only → `install`이 가리키는 파일이 실제로 있어서 다운로드되는지.
4. **거짓 완료 금지**: 위 1~3을 실제로 확인하기 전에는 "등록 완료"라고 보고하지 않는다. 실물(앱.md·설치파일·열린 포트)이 없으면 미완성이다.

## 흔한 실패 → 대응
- 앱 탭에 안 뜸 → `앱.md`에 `description`(또는 `name`)이 비었는지 확인(둘 다 없으면 스캔이 건너뜀). 경로가 `앱/<등급>/<이름>/앱.md`인지 확인.
- web-app이 다른 컴퓨터에서 안 열림 → 서버가 `localhost`에 바인드됨. `0.0.0.0`으로 바꾼다.
- '실행' 눌러도 안 뜸 → `run` 명령의 실행파일 경로가 앱 폴더 기준인지, 실행권한(+x)이 있는지 확인. 로그는 `앱/<...>/.run/app.log`.
- 큰 바이너리를 `기본`에 둠 → 저장소가 무거워짐. `추가`/`대외비`로 옮긴다.
